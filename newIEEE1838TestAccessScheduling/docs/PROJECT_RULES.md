# 项目规则

本文件记录项目长期约束。后续新增代码、实验和文档时优先遵守这里的规则。

## 目录与文件纪律

- 项目根目录只保留入口文档、原始资料和少数顶层配置，不在根目录堆放实验输出。
- 源码只放在 `src/`，实验入口只放在 `experiments/`，测试只放在 `tests/`。
- 可复用输入案例放在 `configs/cases/`。
- 原始外部数据放在 `data/raw/`，清洗或派生数据放在 `data/derived/`。
- 正式结果只放在 `results/` 的分类子目录：`figures/`、`tables/`、`schedules/`、`reports/`。
- 临时文件只能使用 `tmp_*` 或 `scratch/` 命名，并在任务结束前删除或不纳入版本管理。
- 不为每个空目录创建占位文件；只有长期可读的说明、输入、结果才落盘。

## 结果管理

- 不覆盖已有正式结果，除非本次任务明确需要刷新，并在最终回复中说明。
- 每个可用于论文或汇报的结果都应能追溯到输入案例、实验脚本和生成时间。
- 不把失败尝试、调参残留、中间日志长期保存在项目中。
- 如果需要保存调试信息，应汇总成 `results/reports/` 下的一份简短报告，而不是散落多个日志。

## 研究表述边界

- 本项目研究 IEEE 1838-compatible test access resources 上的建模与调度，不声称 IEEE 1838 标准定义调度算法。
- DWR 表示 Die Wrapper Register。
- FPP（Flexible Parallel Port）是 IEEE 1838-2019 标准的可选组件（Clause 7, optional），使用独立物理 lane，通过 TAP 配置。FPP 不是零成本资源，也不是万能控制路径。
- BIST 本地执行阶段可以释放 PTAP，但仍可能占用本地 BIST engine、功耗域和热区域。
- 没有落地代码、输入文件、实验脚本和结果文件支撑的内容，只能写成计划或假设，不能写成已完成结果。

## 当前用户偏好

- 保持目录和输出克制、清晰。
- 后续不要产生大量中间文件、废料文件或散乱结果。
- 优先生成少量可复用、可追溯、能支撑论文的文件。
