# M9: Scenario Expansion

## Goal

M9 demonstrates that the scheduler is not limited to one vertical 3D stack. It
must run three scenario families through the same M1-M8 flow:

- `3d_stack`: the existing 4-die stack baseline.
- `2.5d_interposer`: multiple dies placed side by side on an interposer.
- `5.5d_multi_tower`: multiple vertical towers connected through an interposer.

## Real-Data Rule

M9 inputs must be assembled from `docs/data/M9_DATA_PROVENANCE.md`. A JSON field
can be treated as real data only if its provenance row is marked
`public_measured_or_benchmark` or `derived_from_public_data`.

Fields marked `model_assumption` are allowed only when they are clearly labeled
in the report and, where possible, covered by a sensitivity sweep.

## Scenario-Specific Bottlenecks

| Scenario | Required distinguishing feature | Expected bottleneck |
| --- | --- | --- |
| 3D stack | vertical STAP chain and TSV/micro-bump interconnect tests | serial path, TSV/inter-layer thermal coupling |
| 2.5D interposer | horizontal die placement and interposer route tests | interposer route capacity, horizontal hotspot coupling |
| 5.5D multi-tower | multiple 3D towers on one interposer | tower-internal STAP/TSV plus tower-to-tower route contention |

## Case Construction Policy

The M9 case JSONs should contain only scheduler-required fields. Source metadata
stays in `docs/data/M9_DATA_PROVENANCE.md` to keep the runtime schema stable.

Recommended case files:

```text
configs/cases/2_5d_interposer_m9_public.json
configs/cases/5_5d_multi_tower_m9_public.json
```

Implemented M9 artifacts:

```text
src/model/itc02.py
experiments/generate_m9_cases.py
experiments/run_m9_scenario_suite.py
data/derived/m9_itc02_module_summary.csv
results/tables/m9_scenario_comparison.csv
results/reports/m9_scenario_expansion_report.md
```

The generated cases currently use:

| Case | Benchmark source | Public benchmark fields used | Model-assumption fields |
| --- | --- | --- | --- |
| `2_5d_interposer_m9_public` | ITC'02 `d695.soc` | module I/O, scan-chain lengths, pattern counts | IEEE 1838 bit widths, DWR lengths, per-phase power, thermal RC coefficients |
| `5_5d_multi_tower_m9_public` | ITC'02 `p22810.soc` | module I/O, scan-chain lengths, pattern counts | IEEE 1838 bit widths, DWR lengths, per-phase power, thermal RC coefficients |

Run sequence:

```bash
python experiments/generate_m9_cases.py
python experiments/run_m9_scenario_suite.py --time-limit-s 10
```

The scenario suite writes only a cross-scenario table and report by default. Per-method
schedule CSVs can be requested with `--schedule-dir results/schedules`, but they are
not emitted by default to keep M9 output compact.

Each generated report must state:

- which fields came from public data;
- which fields were derived from public data;
- which fields were model assumptions;
- which outputs were generated by this repository.

## Current M9 Result Snapshot

The latest generated result table compares `pure_serial`, `m4_greedy`, and
`m5_cpsat` on all three scenarios. With a 10 s CP-SAT time limit:

| Case | Best method | Makespan (s) | Normalized vs serial | Solver status |
| --- | --- | ---: | ---: | --- |
| `3d_stack_m1_example` | `m5_cpsat` | 0.003963920 | 0.0621 | OPTIMAL |
| `2_5d_interposer_m9_public` | `m5_cpsat` | 0.001762160 | 0.4235 | FEASIBLE |
| `5_5d_multi_tower_m9_public` | `m5_cpsat` | 0.002455400 | 0.0269 | FEASIBLE |

`FEASIBLE` means CP-SAT found a valid schedule within the time limit but did not
prove global optimality.

## Not The Final Experiment

M9 is a flow-completion milestone, not the final experimental section. It proves
that one 3D stack, one 2.5D interposer case, and one 5.5D multi-tower case can
all pass through the same M1-M8 modeling and scheduling pipeline.

It is intentionally too small for the final paper-scale evaluation. The current
M9 result must not be presented as the full experiment because:

- it uses only two ITC'02 workloads (`d695` and `p22810`) in generated cases;
- it compares only a small number of topology instances;
- it does not sweep FPP lane count, power budget, or thermal coupling strength;
- CP-SAT is only proven optimal for the small 3D baseline under the current time
  limit, while larger generated cases are feasible-but-not-proven-optimal;
- HotSpot-level validation is still outside the current result table.
