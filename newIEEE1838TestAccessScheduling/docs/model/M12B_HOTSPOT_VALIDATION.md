# M12b: HotSpot Remote Validation

## Goal

M12b runs a small offline HotSpot validation on the Linux/EDA VM. It is separate
from the scheduler: schedules are generated by M12, exported as `.flp/.ptrace`,
then sent to the VM for HotSpot execution.

This stage distinguishes three artifacts:

- proxy results: `results/tables/m12_thermal_validation_summary.csv`;
- HotSpot inputs: `results/hotspot/m12/*.flp` and `*.ptrace`;
- HotSpot outputs: `results/tables/m12_hotspot_validation_summary.csv` and
  `results/reports/m12_hotspot_validation_report.md`.

## VM Assumptions

Default VM parameters:

```text
user: autumn
host: 192.168.113.142
key:  C:\Users\ZhuanZ1\.ssh\codex_rhel6
remote work dir: ~/m12b_hotspot_validation
```

The VM must provide:

- `ssh` access from Windows;
- `gcc` and `make`;
- a runnable `hotspot` binary in `PATH` or passed with `--hotspot-bin`.

HotSpot source code and binaries are not stored in this repository.
Source: the UVA HotSpot page points to the `uvahotspot/HotSpot` GitHub
repository (`https://lava.cs.virginia.edu/HotSpot/`,
`https://github.com/uvahotspot/HotSpot`).
The deployed VM copy is outside the repository:

```text
source/binary root: /home/autumn/tools/HotSpot-master
hotspot binary:     /home/autumn/tools/HotSpot-master/hotspot
build option:       make EXTRAFLAGS=-std=gnu99
```

## Commands

Regenerate representative HotSpot inputs:

```powershell
python experiments/run_m12_thermal_validation.py
```

Dry-run input and command validation:

```powershell
python experiments/run_m12_hotspot_remote_validation.py --dry-run
```

Actual remote run:

```powershell
python experiments/run_m12_hotspot_remote_validation.py
```

If the VM uses a custom HotSpot path:

```powershell
python experiments/run_m12_hotspot_remote_validation.py --hotspot-bin /home/autumn/tools/HotSpot-master/hotspot
```

If the HotSpot build requires a config file:

```powershell
python experiments/run_m12_hotspot_remote_validation.py --remote-config /path/to/hotspot.config
```

## Interpretation Rules

- Rows with `status=ok` are HotSpot execution evidence.
- Rows with `dry_run`, `skipped`, `missing_input`, or `failed` are not validation
  evidence and should only be used for debugging.
- The current floorplan is block-level and simplified. It supports trend checks
  against the proxy model, not thermal signoff.

## Current M12b Status

The implementation is complete and the representative input set contains 6 rows:

- 3 cases: small 5.5D multi-tower, medium 3D stack, medium 5.5D multi-tower;
- 2 schedules per case: `thermal_min_risk` and `m4_greedy`.

The first remote run against the old VM address did not execute HotSpot because
the VM was unreachable:

```text
ssh: connect to host 192.168.113.131 port 22: Connection timed out
```

The target was then updated to `autumn@192.168.113.142`. The runner uses
noninteractive SSH with a disposable known-hosts path for this lab VM so
recycled VM host keys do not block the experiment. The SSH key is accepted by
the VM, and the VM provides `gcc`, `make`, `wget`, `curl`, `tar`, and `unzip`.

```text
ssh -i C:\Users\ZhuanZ1\.ssh\codex_rhel6 autumn@192.168.113.142
```

HotSpot was not preinstalled on the VM. It was deployed from the official
`uvahotspot/HotSpot` source archive into `/home/autumn/tools/HotSpot-master`
and built with `make EXTRAFLAGS=-std=gnu99` because the VM compiler defaults to
an older C mode.

Generated files:

```text

results/reports/m12_hotspot_validation_report.md
```

Rows are valid HotSpot evidence only after the report says `HotSpot executed:
yes`. If rows remain `status=skipped` or `status=failed`, cite the recorded
reason as deployment/debug information rather than thermal validation evidence.
