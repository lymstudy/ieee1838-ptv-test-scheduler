## 1. 四篇参考文献关键方法总结

### 1.1 Habiby 2020

**论文**：*Power-aware Test Scheduling for IEEE 1687 Networks with Multiple Power Domains* (IEEE DFT 2020, 6 pages)

**核心问题**：IEEE 1687 (IJTAG) 可重构扫描网络中，多个电源域约束下的测试调度。

#### 方法要点

**1) 图建模**

- 将 ICL 网表解析为有向图 G(V, E)
- SCB / ScanMux 各对应一个节点
- SIB 拆为两个节点（a/b，分别对应 bypass 和接入两种状态）
- 顶点权重 = 该节点寄存器的位数
- 边权重 = 两端顶点权重的平均值（将顶点权重转化为边权重以适配 Dijkstra）

**2) 最短路径延迟计算（Algorithm 1）**

- 使用 Dijkstra 从 SI 出发求最短 write path，从 SO 出发（图反向）求最短 read path
- Write latency + Read latency = 该 instrument 的单次访问时间
- 只需运行两次 Dijkstra（SI 出发 + SO 出发），即可得到所有 instruments 的延迟
- 可以识别 critical path（最大延迟的 instrument）

**3) 并发组识别**

- BFS 遍历图，每个节点维护一个 "并发表"（记录从 SI 到该节点所有路径上可被同时访问的 instruments）
- 汇聚节点（如 ScanMux）将来自不同路径的并发集合合并（"|" 分隔）
- 最终在 SO 节点的并发表就是网络结构层面的最大并发组合

**4) 功率约束检查（二叉决策树 + 剪枝）**

- 对并发组内 instruments 枚举所有子集
- 每加入一个 instrument 检查是否超出该 domain 的 power limit
- 一旦超限立即剪枝
- 区分 **concurrent access**（同一条扫描链上 instruments 同时执行）和 **sequential access**（不同链顺序访问但可能存在 execution time 重叠）

**5) 实验**：Mingle, BasicSCB, TreeFlat, q12710, N17D3, N32D6, H953 共 7 个 ITC'16 benchmark，报告了并发路径数、最大/最小 write/read latency。

#### 与本研究的关联

- 图建模方法与本研究的"异构测试访问图"一致，但 Habiby 只有 SIB/ScanMux，本研究有 PTAP/STAP/FPP/DWR 多种资源
- 并发组识别直接对应 Test Access Recipe 中哪些 recipe 可以同时执行
- 功率剪枝搜索可简化为 CP-SAT 中的 power domain 约束
- **应作为 baseline 实现**

### 1.2 Habiby 2022

**论文**：*Power-aware test scheduling framework for IEEE 1687 multi-power domain networks using formal techniques* (Microelectronics Reliability, 2022, 10 pages)

这是四篇中**方法最成熟、与本研究 CP-SAT + ALNS 框架最相关**的一篇。

#### 方法要点

**1) IJTAG 网络结构的 SAT/CNF 建模**

三类基本结构的 CNF 表达：

| 结构 | CNF 公式 | 含义 |
|---|---|---|
| SIB（开关） | `Φ = s ∨ ¬x` | SIB 关闭时子网络不可达 |
| 分支节点（n 个后继） | `Φ = (¬x₁ ∨ x₂ ∨ … ∨ xₙ) ∧ (∧_{i≠j} ¬xᵢ ∨ ¬xⱼ) ∧ (∧_{i=2}^n x₁ ∨ ¬xᵢ)` | 仅一条活跃链；源节点激活则必有后继激活；后继激活则源节点必激活 |
| 汇聚节点/Mux（n 个前驱） | `Φ = (x₁ ∨ … ∨ xₙ₋₁ ∨ ¬xₙ) ∧ (∧_{i≠j} ¬xᵢ ∨ ¬xⱼ) ∧ (∧_{i=1}^{n-1} xᵢ ∨ ¬xₙ)` | 多输入不同时激活；任一输入激活则输出激活；输出激活则必有一输入激活 |
| 简单链 | `Φ = x₁ ↔ x₂ = (x₁ ∨ ¬x₂) ∧ (¬x₁ ∨ x₂)` | 相邻节点状态相同 |

**2) PBO 增量局部优化**

- 每 session 最大化 concurrent active instruments
- 目标函数：`F = Σ pᵢ · xᵢ`（以功耗为权重，优先排满高功耗 instruments）
- 逐步覆盖：每轮从 pool 中移除已完成足够访问次数的 instruments
- 最多 N 轮迭代（N = instrument 数量），上界保证
- 局部最优解的质量：与 ILP 全局最优的 MAPE < 10%

**3) ILP 全局优化**

- 引入 n × n 决策矩阵 M：行 = session，列 = instrument
- `xᵢⱼ ∈ {0, 1}`：instrument j 在 session i 中是否活跃
- `aᵢ`（整数变量）：session i 的 CSU（Capture-Shift-Update）周期数
- 目标函数：`Min Σ aᵢ`（最小化总 CSU 数）
- 三个约束：网络结构约束（CNF → ILP 不等式）、功率约束（每个 domain 求和 ≤ Pmax）、访问覆盖约束

**关键技巧 — Glover's 线性化方法**：

访问约束 `Σ aᵢ · xᵢⱼ = Aⱼ` 是非线性的（两个决策变量相乘），通过引入 `yᵢⱼ = aᵢ · xᵢⱼ` 并添加线性不等式来线性化：

```
0 ≤ yᵢⱼ ≤ A · xᵢⱼ
aᵢ - A(1 - xᵢⱼ) ≤ yᵢⱼ ≤ aᵢ
```

其中 A 是所有 instruments 中最大所需访问次数。

**4) 实验结果**

| 指标 | PBO (clasp) | ILP (CPLEX) |
|---|---|---|
| 适用范围 | 1000+ instruments | 中小网络（<800 CNF clauses） |
| 解的质量 | 局部最优，MAPE < 10% | 全局最优 |
| Runtime | 秒级 | 分钟到小时级 |
| 最大网络 | NE1200P430（1629 instruments） | 受限于 CNF 规模 |

#### 与本研究的关联

**直接对应关系**：

| Habiby 2022 概念 | 本研究的对应 |
|---|---|
| Instrument | Test Target（每 die 上的测试对象） |
| Session（一组 CSU） | Test Access Recipe 的一次执行 |
| CNF 网络结构约束 | PTAP/STAP 串行互斥约束 |
| Power domain 约束 | 电热功耗上限约束 |
| PBO 增量优化 | M4 Greedy 贪心调度 |
| ILP 全局优化 | M5 CP-SAT 精修 |
| Glover 线性化 | 可用于本研究的 recipe × duration 非线性项处理 |

**必须实现为对比基线。**

### 1.3 Patmanathan 2024

**论文**：*Thermal-Aware Test Scheduling with Floorplanning for Three-Dimensional Stacked Integrated Circuit* (IEEE ICSE 2024, 4 pages)

**核心问题**：3D-SIC 的热感知测试调度 + 布局规划（floorplanning）协同优化。

#### 方法要点

**1) Pareto Test Cubes**

- 每个 core 有多个 wrapper 配置（不同 TAM width → 不同 test time、不同 peak power）
- 只保留 Pareto-optimal 配置
- 每个 Pareto cube 是三维的：TAM width（wᵢⱼ）× Test time（T(wᵢⱼ)）× Peak power（P(wᵢⱼ)）

> **这与本研究的 Test Access Recipe 高度对应！** Recipe = 对一个 die/instrument 的多种访问方式选择（BIST/FPP/Hybrid/Serial），cube 的三个维度 = recipe 的访问时间/功耗/资源占用。

**2) 蚁群优化 (ACO)**

- 每只蚂蚁为每个 core 选一个 Pareto cube
- 信息素 `τ(i, j)` + 启发式偏好 `Wprefer`（偏向最大 TAM width = 最短 test time）
- 概率选择公式：`prob(i,j) = [τ(i,j)]^α × [wᵢⱼ]^β / Σ(...)`
- `Rglobal`：全局最优蚂蚁的 cube 集合，用于更新 `Wprefer`
- 迭代至最大循环次数或收敛（所有蚂蚁选相同 cube）

**3) 热感知 3D 布局规划**

- 识别 hot module（power density > 平均值的 core）
- 三条布局规则：
  1. Hot modules 不邻接
  2. Hot module 靠近 heat sink
  3. Hot module 靠近层边缘（增加空气接触散热）
- 权衡函数：`TFᵢ = γ · wᵢⱼ + δ · pᵢⱼ`（平衡 TAM width 需求与 power density）

**4) RHDF（相对散热因子）**

```
RHDF = HHDF × VHDF × Power trace
```

- **HHDF**（水平散热函数）= core 到层四边水平距离之和
- **VHDF**（垂直散热函数）= 层到 heat sink 的垂直距离
- 调度时按 RHDF **降序**排列（散热能力差的 core 优先调度，因为它们需要更多散热空间/时间）

**5) 邻接排斥原则 (Adjacency Exclusion)**

- 同一 test session 内，**不在 3D 布局中相邻的 core 才可并发测试**
- 通过为被测 core 周围留出散热空间来防止热点叠加

**6) 3D Bin Packing 调度**

- 输入：cube 集合 + RHDF 排序列表 + Wmax/Pmax/Tmax
- 流程：选 cube（按 RHDF 排序）→ 检查邻接排斥 → 检查 TAM width 余量 → 检查总功率 ≤ Pmax → HotSpot 6.0 检查温度 ≤ Tmax
- 任一约束不满足 → 放入临时列表，等待下一 test session

**7) 实验**：ITC'02 d695, p22810, p34392, p93791 的 3D-SIC 版本，有/无 floorplanning 对比，TAM width 从 16 到 56 扫描。改善约 0.2%（TAT 差异极小）。

#### 与本研究的关联

| Patmanathan 2024 概念 | 本研究的对应 | 可借鉴之处 |
|---|---|---|
| Pareto cube | Test Access Recipe | Recipe 的 Pareto 剪枝（M3 已做） |
| ACO 蚁群选择 | ALNS 外层搜索 | 用 ACO 替代/补充 ALNS 的 recipe 选择 |
| RHDF | 热代理模型的 input feature | 显式计算散热因子加入调度 score |
| 邻接排斥 | "热邻接冲突约束"（摘要中提及） | 编码为 CP-SAT 约束 |
| HotSpot 6.0 离线验证 | M12b HotSpot 验证 | 仅用于温度上限检查，不参与优化闭环 |

**注意**：Patmanathan 2024 的实验改善仅 0.2%，这是该领域方法差异小的正常现象。不应期望本方法在普通 benchmark 上有大幅改善。

### 1.4 Sen Gupta 2011

**论文**：*Scheduling Tests for 3D Stacked Chips under Power Constraints* (IEEE DELTA 2011, 6 pages)

**核心问题**：3D-SIC 的 pre-bond (wafer test) 和 post-bond (final test) 测试时间协同优化，考虑功耗约束和 BIST 控制线数量。

#### 方法要点

**1) 三种策略**

| 策略 | 知识前提 | 操作 | 效果 |
|---|---|---|---|
| **Serial Processing (SP)** | 仅知每个 chip 的 pre-bond test time | Post-bond 阶段将两个 chip 的测试**串行**执行 | Baseline，TAT 最大 |
| **Partial Overlapping (PO)** | 知每个 session 的峰值功耗和 test time | 将不同 chip 中**功耗兼容**的 session 在 post-bond 并置 | TAT 减少，不拆分 session |
| **ReScheduling (RS)** | 知每个 test 的 duration + power | 将 pre-bond session **拆分重组**，使 post-bond 阶段更多测试并置 | TAT 最大减少，但可能增加 BIST 控制线 |

**2) ReScheduling 两步法**

**Step 1 — Session Pair 重组**：

- 从两个 chip 各取一个 session（Sx, Sy）
- 将其中的 tests 按 duration 降序排列到列表 M
- 贪心地将最长测试放入新 session Sa，直到 Sa 总功耗超出 Pmax
- 剩余测试自动归入 session Sb
- 按 (Sa, Sb) 重新分配两个 chip 各自的 pre-bond + post-bond 测试
- 计算 TAT 减少量，记录到 session pair 矩阵

**Step 2 — 全局 Session Pair 选择**：

- 将 TAT 减少量填入矩阵：行 = ChipY sessions，列 = ChipX sessions
- 贪心选择：行列排序后，每次选最大减少量的 session pair，然后删掉该行该列
- 选出的 session pairs 形成最终的 pre-bond + post-bond schedule

**3) BIST 控制线 Trade-off**

- 拆分 session 会增加 BIST 控制线数量
- 最终 schedule 由设计者根据 TAT reduction vs. 控制线增加做权衡

**4) 实验结果**

- Benchmark：ASIC Z (9 cores), System L (14 cores), Muresan (10 cores) 的两两组合形成 3D-SIC
- 最大 TAT 减少：17.1%（System L + System L）
- 平均 TAT 减少：7.7%（6 组实验）
- 其中 4 组不需要拆分 session（不增加控制线），2 组需要拆分
- 不需要拆分时，PO 和 RS 结果相同

**5) 复杂度分析**

- Step 1：O(N log N)，N = 两个 session 中 tests 总数
- Step 2：O(T log T)，T = session pair 矩阵元素数
- 总复杂度：O(T log T)

#### 与本研究的关联

| Sen Gupta 2011 概念 | 本研究的对应 |
|---|---|
| Pre-bond wafer test | 独立 die 的 BIST self-test |
| Post-bond final test | Stacked system 的联合测试调度 |
| Session（一组同时执行的 tests） | Test session（一组同时执行的 recipes） |
| Session 拆分重组 | ALNS 的 destroy/repair 操作 |
| 控制线数量 trade-off | FPP lane 占用 / PTAP 占用 trade-off |

**应实现为对比基线。** 尤其是 Partial Overlapping 策略——简单、有效、有理论复杂度分析。

## 3. 本实验当前状态诊断

### 3.1 致命问题：实验自己推翻了核心宣称

**M11 算法对比结果（6 个 small/medium 场景，8 FPP lanes，nominal power）：**

| 方法 | 平均归一化完成时间 | 平均加速比 | 备注 |
|---|---|---|---|
| pure_serial | 1.0000 | 1.00 | 串行基线 |
| fixed_fastest | **0.1819** | 22.80 | 先定最快路径，再调度 |
| tam_like | **0.1819** | 22.80 | 简化 TAM 装箱 |
| m4_all_recipes | **0.1819** | 22.80 | M4 贪心（无 Pareto 剪枝） |
| m4_greedy | **0.1819** | 22.80 | M4 贪心（Pareto 剪枝后） |
| m5_cpsat | 0.1814 | 22.86 | CP-SAT 精修 |
| m6_alns | 0.3394 | 2.95 | ALNS（更差） |

**fixed_fastest = tam_like = m4_all_recipes = m4_greedy 完全相同（0.1819）**。

这意味着：**在普通 benchmark 上，「先定最快路径再做调度」和「路径-调度联合优化」产生了一模一样的结果。** 论文摘要中宣称的「避免传统先定路径、再做调度带来的局部受限问题」——在数据上没有体现。

**M22 消融实验的自我证伪：**

| 消融条件 | vs fixed-fastest 增益 | 说明 |
|---|---|---|
| m10_original_control | **0.00%** | 普通 M10 benchmark |
| bist_private_control | **0.00%** | 有压力但无共享瓶颈 |
| shared_bist_no_parallel_escape | **0.00%** | 有共享瓶颈但无替代路径 |
| shared_bist_with_parallel_escape | **25.84%** | 共享瓶颈 + 替代路径共存 |

前三行全是 0%。方法的有效性仅在最末一个人工构造的 "shared BIST pressure + FPP alternative" 条件下才显现。

### 3.2 次要问题

**a) 热模型是装饰品**

| 指标 | M7 代理模型 | HotSpot 6.0 |
|---|---|---|
| d695 5.5D peak temp | 25.09 °C | 60.79 °C |
| p22810 3D peak temp | 25.12 °C | 60.61 °C |
| 跨方法温差 | ~0.1 °C | ~2 °C |

- 一阶 RC 代理模型给出的绝对温度完全不可信（比环境温度高 <0.5 °C）
- 所有 scheduler 的 peak temperature 差异仅在 0.1 °C 量级——热约束实际上没有生效
- HotSpot 只作为离线一次性验证（6 行），不参与优化闭环
- 摘要中宣称的「热风险评估与温度反馈机制」名不副实

**b) ALNS 不是真正的 ALNS**

- 自适应权重更新是 ALNS 的定义性特征，但当前实现是 **round-robin 循环**
- M11 中 medium 场景 ALNS **被跳过**（target 数 > 16）
- ALNS 结果（0.3394）比所有方法（除 pure_serial）都差
- M17 创新支撑审计评级："weak"
- M21 明确写："ALNS still needs a stronger implementation before becoming a core contribution"

**c) Benchmark 数据全是模型假设**

- 所有数据来源：ITC'02（2002 年 2D SoC benchmark），手工映射到 2.5D/3D/5.5D
- 功耗、热阻、热容、FPP 带宽等参数标签为 `model_assumption`
- 没有真实 Chiplet/HBM/interposer/3D 堆叠芯片的测试数据
- 所有"加速比"都在自己设定的参数空间内循环

**d) CP-SAT 从未证明最优性**

- M11 solver status 全部是 **FEASIBLE**，没有一行 **OPTIMAL**
- 没有 optimality gap 分析
- 无法区分 CP-SAT 比 M4 好 0.0005 是因为算法好还是搜索运气好

**e) 没有实现任何文献方法作为对比基线**

M11 对比的 8 个方法全是自己写的（pure_serial, fixed_fastest, tam_like, low_power, m4, m5, m6）。没有实现 Habiby 2022 的 PBO、Sen Gupta 2011 的 PO/RS 等 published baseline。

**f) B 阶段（IEEE 1838 分层调度）核心未完成**

- STATUS.md 明列 B3-B10 全部未完成
- B3.1 只是 deterministic earliest-start list scheduler，没有 PTAP/STAP 细粒度建模
- 「3DCR select/bypass、STAP path opening、DWR mode/shift/capture/update 的细粒度行为」明确列为 _尚未建模_

**g) 实验叙事过于防御性**

项目文档中频繁出现：
- "Do not claim this gain on ordinary M10 sweep"
- "These rows are not global optimality claims"
- "Still a research model, not automatic DFT extraction"
- "Do not write: Joint scheduling is better on all real benchmarks"

这些限定词虽然诚实，但暴露了实验撑不起论文宣称的根本问题。

## 4. 具体改进方案

### 4.1 P0（必须做）：实现文献对比基线

**目标**：在相同的 benchmark 上，与至少 2 篇已发表论文的方法进行公平对比。

#### Baseline 1：Habiby 2022 PBO 增量调度

- 将 Test Access Recipe 的选择问题编码为 SAT/CNF：
  - 每个 recipe 对应一个布尔变量 `xᵢ ∈ {0,1}`
  - 互斥约束（PTAP 串行）= 分支节点 CNF
  - 资源容量约束（FPP lane）= PBO 求和约束 `Σ lanesᵢ · xᵢ ≤ max_lanes`
  - 功耗约束 = `Σ powerᵢ · xᵢ ≤ Pmax`
- 目标函数：`F = Σ weightᵢ · xᵢ`（最大化当轮覆盖的 test targets）
- 使用开源 clasp solver（https://github.com/potassco/clasp）
- 增量求解：每轮覆盖一部分 targets，从 pool 中移除，继续下一轮
- 报告：TAT、runtime、session 数量、与 ILP/CP-SAT 的 gap

#### Baseline 2：Sen Gupta 2011 Partial Overlapping

- 将每个 die 的 pre-bond schedule 作为一个独立 session
- Post-bond 阶段：检查不同 die 的 session 之间是否可以功率兼容地并置
- 贪心匹配最大的可并置 session pair
- 报告：TAT vs. Serial Processing baseline 的减少量

#### Baseline 3（可选）：Patmanathan 2024 ACO

- 实现简化的 ACO 选择 recipe
- 对比 ACO 选 recipe vs. M4 greedy 选 recipe 的效果差异

### 4.2 P0（必须做）：重新设计 Benchmark 实验

**目标**：让方法差异在合理的场景中显现，而不是用无效场景。

**策略**：基于 M22 的发现，将 benchmark 分为两类：

**A 类 — 规模与覆盖验证**：
- 使用原有 M10 普通 benchmark
- 用途：展示方法在多种规模/拓扑下**能跑**，不用于宣称方法优势
- 只报告 serial vs. M4 greedy 的加速比（不要求 M4 > fixed_fastest）

**B 类 — 机制验证**：
- 使用 M21/M22 的 resource pressure suite
- 论证 shared resource pressure 在 3D-IC 中的普遍性：
  - 多个 die 共享有限 TSV 做 BIST 访问（IEEE 1838 的根本特征）
  - 多个 die 共享 FPP lanes（物理 lane 数有限）
  - 热邻接冲突（3D 堆叠中 die-to-die 热耦合不可避免）
- 在 B 类 benchmark 上展示：fixed_fastest vs. M4 vs. M5 的显著差异
- 区分方法差异是"机制驱动"还是"参数巧合"

### 4.3 P1（强烈建议）：修复或放弃 ALNS

**选项 A — 修复 ALNS**（推荐，如果时间允许）：
- 实现自适应权重更新：`w_d = λ·w_d + (1-λ)·score`（基于 destroy/repair 的历史表现）
- 扩大 ALNS 支持范围（当前 max_alns_targets=16 太受限）
- 在 M21 pressure suite 上展示 ALNS 能改善 CP-SAT feasible 解

**选项 B — 放弃 ALNS 作为核心贡献**（如果时间不够）：
- 论文贡献改为 "CP-SAT based constrained optimization"
- ALNS 降级为附录中的 "extension prototype"
- 删除摘要中的 "与 ALNS 相结合" 表述

### 4.4 P1（强烈建议）：升级热模型

**最小可行改进**：
- 实现 die-to-die thermal coupling（参考 Patmanathan 2024 的 VHDF 概念，但用 RC 网络连接相邻 die）
- 确保代理模型产生有意义的温度差异（至少 5-10 °C 跨方法差异，而非当前 0.1 °C）
- 在 B 类 benchmark 上展示热感知调度确实降低了峰值温度

**改进方案**：
- 计算每 die 的等效 RHDF（水平 × 垂直散热能力）
- 将 RHDF 作为 CP-SAT 软约束或目标函数的惩罚项
- 在 HotSpot 离线验证中增加 case 数量（从当前 6 行扩展到 20+ 行）

### 4.5 P2（建议）：补 B 阶段核心建模

**至少完成以下内容**：
- PTAP/STAP 串行级联的配置时间模型（参考 Habiby 2020 的 Dijkstra 延迟计算）
- DWR 的 shift/capture/update 阶段细分
- 将 AccessPath 的时间估算校准到至少一个已知参考值（如从 Tessent 文档或公开论文中找到的典型配置时间）

### 4.6 P2（建议）：补充 CP-SAT Optimality Gap 分析

- 对 M5 的 small 场景，让 CP-SAT 跑到 OPTIMAL（增加 time limit）
- 对 medium 场景，报告 best bound 和 optimality gap
- 绘制 gap vs. problem size 曲线

## 5. 修订后的实验章节大纲

```
第 X 章 实验评估

X.1 实验设置
  - Benchmark 来源与特征（ITC'02 → 多拓扑映射，两类别：覆盖验证 / 机制验证）
  - 实现环境（Python 版本、OR-Tools 版本、CP-SAT 参数、clasp/CPLEX 版本）
  - 对比方法列表：
    · 纯串行 IEEE 1838 基线
    · fixed_fastest（定路径+调度）
    · Habiby 2022 PBO 增量调度 [文献基线]
    · Sen Gupta 2011 Partial Overlapping [文献基线]
    · M4 贪心（本方法基础版）
    · M5 CP-SAT（本方法精确版）
    · CP-SAT 全局优化（仅在 small 场景报告 OPTIMAL gap）
  - 评估指标：归一化 makespan、peak power、peak temperature、FPP 利用率、runtime

X.2 规模与拓扑覆盖实验（A 类 benchmark）
  - 216 行 M10 sweep 结果
  - 不同拓扑的加速比分布
  - 目的：证明方法可扩展迁移到 2.5D/3D/5.5D

X.3 算法对比与消融分析（B 类 benchmark）
  - M21 pressure suite 上的全方法对比
  - 与 Habiby 2022 PBO、Sen Gupta 2011 PO 的对比
  - M22 消融：展示"共享瓶颈 + 替代路径"条件下方法差异的来源
  - Pareto 剪枝的效果：m4_all_recipes vs. m4_greedy
  - Optimality gap 分析（small 场景）

X.4 资源敏感性分析
  - FPP lane 扫描（4/8/16/32 lanes）
  - 功耗上限扫描（tight/nominal/relaxed）
  - 热上限扫描
  - 各方法在不同资源压力下的表现变化

X.5 热分析
  - 代理模型 + RHDF 热风险排名
  - HotSpot 6.0 离线验证（20+ 行代表性 case）
  - 代理模型 vs. HotSpot 的排名一致性
  - 局限性讨论（代理模型精度、HotSpot 仅代表性验证）

X.6 讨论与局限性
  - 本方法的适用条件（shared resource pressure + alternative paths）
  - Benchmark 数据的模型假设性质
  - CP-SAT 可扩展性（大规模场景的 feasibility gap）
  - 热模型的简化
  - 与工业工具（Tessent, RedHawk, Voltus）的关系
```
